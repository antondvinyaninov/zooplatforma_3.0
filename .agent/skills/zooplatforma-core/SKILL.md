# Архитектурные правила Zooplatforma 3.0

При работе над проектом Zooplatforma 3.0 агент ОБЯЗАН строго соблюдать следующие правила:

## 1. Роутинг и Кабинеты (org vs orgs)
- **`/orgs`** (множественное число) — это публичный справочник организаций (`app/main/(main)/orgs`).
- **`/org`** (единственное число) — это приватный кабинет/рабочее пространство организации (`app/org/`).
- **`/gov`**, **`/owner`**, **`/pethelper`**, **`/admin`** — соответствующие приватные кабинеты.
- **ЗАПРЕЩЕНО** нарушать этот принцип разделения (plural = публичный список, singular = кабинет) при добавлении новых страниц.

## 2. Платформенная система модулей
- Подключаемые модули (Питомцы, Заявки, Медкарты) — это переиспользуемые функциональные единицы.
- **СТРОГОЕ ПРАВИЛО:** ВЕСЬ функционал модуля (компоненты, страницы, хуки, типы) должен храниться **ТОЛЬКО** внутри `components/modules/[id]/`. Обязательная структура: `list/`, `profile/`, `settings/`, `shared/`, `widgets/`.
- Файлы `page.tsx` внутри кабинетов (например, `app/org/page.tsx`) — это ТОЛЬКО тонкие точки входа (маршруты). Они должны состоять из 5-8 строк, импортируя готовый компонент из `components/modules/`.
- Работает двухуровневый доступ: глобальный (`module_cabinet_permissions`) и локальный активатор пользователя.

## 3. UI-компоненты и принцип "Один файл — один функционал"
- Общие компоненты не дублируются по кабинетам, а лежат в общей папке платформы (например, `frontend/components/pets/`).
- **ЗАПРЕЩЕНО** размещать несколько независимых визуальных блоков (модалка + таблица + фильтры) в одном файле. Если файл больше ~200 строк — разбиваем его на мелкие. `PetsTable.tsx` отвечает только за таблицу.

## 4. Базы данных и миграции
- **ЗАПРЕЩЕНО** использовать `DROP TABLE` или массовые `UPDATE` (особенно касающиеся колонок вроде `password_hash`), которые перезаписывают старые данные, без многократного явного подтверждения.
- Использовать только безопасные операции, такие как `ADD COLUMN IF NOT EXISTS`.
- При выполнении команд через терминал всегда проверять возможность создания дампа или бэкапа перед запуском `golang-migrate`.

## 5. Правило общения
- Вся документация, `TaskSummary`, файлы планов (`task.md`, `walkthrough.md`, `implementation_plan.md`) и ответы пользователю ДОЛЖНЫ быть **на русском языке**.

## 6. Ведение Changelog и Обновлений по команде "запиши"
- Агент ОБЯЗАН вести учет изменений в двух файлах проекта (`CHANGELOG.md` для разработчиков и `RELEASES.md` для пользователей).
- **ВАЖНО:** Агенту ЗАПРЕЩЕНО самостоятельно (без явной просьбы) обновлять эти файлы после каждой мелкой задачи, чтобы не тратить токены и время.
- Обновление логов происходит **ТОЛЬКО** когда пользователь явно пишет команду **"запиши"** или прямо просит обновить ченджлог.
- Если дана команда "запиши", агент должен проанализировать историю текущей беседы или последний блок задач, и добавить соответствующие записи в оба файла.

## 7. UI/UX и Мобильный Дизайн (Premium Mobile-First)
- **Fluid Media:** Использовать нативные теги `<img />` с `object-fit: contain` и ограничителями высоты (`max-h-[600px]`), агрессивно избегать жестких `aspect-square` или `bg-[url()]`, чтобы не обрезать оригинальные пропорции фото.
- **Цветные плашки (Chips/Pills):** Сухие списки (компетенции, статусы собак, пол) ВСЕГДА выводить в виде ярких цветных тегов-стикеров с мягкими фонами (напр. `bg-blue-50 text-blue-700 px-3 py-1 rounded-full`).
- **Скругления и тени (Unified Card UI):** Для всех базовых структурных блоков (карточки постов, профили питомцев, виджеты, боковые панели) СТРОГО использовать единый стандарт скругления `rounded-lg` (8px), тонкую обводку `border-gray-200` и мягкую тень `shadow-sm` (при наведении `hover:shadow-md`). Отказаться от чрезмерных `rounded-2xl` или `rounded-3xl` для главных контейнеров, чтобы интерфейс выглядел однородным. Внутренние кнопки и плашки могут быть `rounded-md` или `rounded-full`.
- **Обрезание текста (Smart Truncation):** Для длинных описаний использовать JS слайсинг (`text.slice(0, 350)`) с кнопкой "Показать полностью...", а не CSS `line-clamp`, чтобы избежать багов с пустым пространством на мобилках.
- **Mobile-First Иерархия:** На телефонах (`max-md:`) сначала идут крупные фото (галереи на 100% ширины) и ключевые кнопки-действия. Тяжелые таблицы, списки постов и настройки смещаются в самый низ экрана.
