# Database Migration Rules

When creating or modifying database migration files (e.g. `*.up.sql` or `*.down.sql`), the Agent MUST strictly adhere to the following rules to prevent data loss (such as accidental deletion of the `password_hash` column):

1. **Только `ALTER TABLE` с безопасными инструкциями:**
   Используйте команды вида `ADD COLUMN IF NOT EXISTS`, оставляя существующие данные (например, `password_hash`) нетронутыми. Не используйте `DROP TABLE` без явного на то, многократного указания пользователя.

2. **Не используем `DEFAULT` для перетирания старых данных:**
   Никогда не делайте массовые `UPDATE users SET column_name = ...` в файлах миграций для всех пользователей разом, если это может затереть их личную информацию или хэши паролей.

3. **Бэкапы (Резервное копирование):**
   При выполнении команд миграции через терминал (или перед запуском `golang-migrate`), всегда предлагайте или создавайте дамп (копию) базы данных, чтобы иметь возможность откатиться назад в случае непредвиденных ошибок.

# Язык общения и документов (Language Rules)

Все коммуникации с пользователем, а также системные артефакты (включая, но не ограничиваясь: `task.md`, `implementation_plan.md`, `walkthrough.md`) ДОЛЖНЫ быть написаны на **понятном русском языке**. Это также касается процесса обдумывания (мыслей), составления резюме для пользователя (TaskSummary) и статусов работы (TaskStatus). Пишите все ответы, пояснения и документацию на русском языке.

# Архитектура URL-маршрутов (Frontend Routing)

## Публичные страницы (Next.js rewrites → `app/main/(main)/`)

Все публичные страницы платформы живут в `frontend/app/main/(main)/` и проксируются через `next.config.ts` rewrites:

| URL | Внутренняя папка | Назначение |
|---|---|---|
| `/` | `main/(main)/` | Лента (главная) |
| `/orgs` | `main/(main)/orgs/` | 📋 Справочник всех организаций |
| `/orgs/:id` | `main/(main)/orgs/[id]/` | 📄 Публичная страница организации |
| `/catalog` | `main/(main)/catalog/` | Каталог животных |
| `/pets/:id` | `main/(main)/pets/[id]/` | Страница питомца |
| `/profile` | `main/(main)/profile/` | Профиль пользователя |

> ⚠️ **ВАЖНО:** `/org` — это НЕ публичный справочник. `/org` = кабинет организации (см. ниже). Справочник — только `/orgs`.

## Кабинеты пользователей B2C (личные, `app/`)

| URL | Папка | Кабинет |
|---|---|---|
| `/owner` | `app/owner/` | 👤 Кабинет владельца питомца |
| `/pethelper` | `app/pethelper/` | 🤝 Кабинет зоопомощника (волонтёра) |

## Кабинеты организаций B2B/B2C (`app/`)

| URL | Папка | Кабинет |
|---|---|---|
| `/org` | `app/org/` | 🏢 Кабинет организации (приют, клиника, магазин, заводчик...) |

Тип организации определяет набор активных модулей (permissions). Один кабинет — разные модули.

## Государственные кабинеты B2G (`app/`)

| URL | Папка | Кабинет |
|---|---|---|
| `/gov` | `app/gov/` | 🏛️ Кабинет государственного/контролирующего органа |

## Служебные кабинеты

| URL | Папка | Кабинет |
|---|---|---|
| `/admin` | `app/admin/` | ⚙️ Панель администратора платформы |

## Принцип разделения org vs orgs

- **`/orgs`** (plural) = публичный **список** всех организаций (REST-конвенция: plural для коллекции)
- **`/org`** (singular) = **рабочее пространство** конкретной организации (кабинет)

Этот принцип НЕЛЬЗЯ нарушать при добавлении новых страниц.
