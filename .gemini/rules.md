# Database Migration Rules

When creating or modifying database migration files (e.g. `*.up.sql` or `*.down.sql`), the Agent MUST strictly adhere to the following rules to prevent data loss (such as accidental deletion of the `password_hash` column):

1. **Только `ALTER TABLE` с безопасными инструкциями:**
   Используйте команды вида `ADD COLUMN IF NOT EXISTS`, оставляя существующие данные (например, `password_hash`) нетронутыми. Не используйте `DROP TABLE` без явного на то, многократного указания пользователя.

2. **Не используем `DEFAULT` для перетирания старых данных:**
   Никогда не делайте массовые `UPDATE users SET column_name = ...` в файлах миграций для всех пользователей разом, если это может затереть их личную информацию или хэши паролей.

3. **Бэкапы (Резервное копирование):**
   При выполнении команд миграции через терминал (или перед запуском `golang-migrate`), всегда предлагайте или создавайте дамп (копию) базы данных, чтобы иметь возможность откатиться назад в случае непредвиденных ошибок.

# Язык общения и документов (Language Rules)

Все коммуникации с пользователем, а также системные артефакты (включая, но не ограничиваясь: `task.md`, `implementation_plan.md`, `walkthrough.md`) ДОЛЖНЫ быть написаны на **понятном русском языке**. Это также касается процесса обдумывания (мыслей), составления резюме для пользователя (TaskSummary) и статусов работы (TaskStatus). Пишите все ответы, пояснения и документацию на русском языке.

# Архитектура URL-маршрутов (Frontend Routing)

## Публичные страницы (Next.js rewrites → `app/main/(main)/`)

Все публичные страницы платформы живут в `frontend/app/main/(main)/` и проксируются через `next.config.ts` rewrites:

| URL | Внутренняя папка | Назначение |
|---|---|---|
| `/` | `main/(main)/` | Лента (главная) |
| `/orgs` | `main/(main)/orgs/` | 📋 Справочник всех организаций |
| `/orgs/:id` | `main/(main)/orgs/[id]/` | 📄 Публичная страница организации |
| `/catalog` | `main/(main)/catalog/` | Каталог животных |
| `/pets/:id` | `main/(main)/pets/[id]/` | Страница питомца |
| `/profile` | `main/(main)/profile/` | Профиль пользователя |

> ⚠️ **ВАЖНО:** `/org` — это НЕ публичный справочник. `/org` = кабинет организации (см. ниже). Справочник — только `/orgs`.

## Кабинеты пользователей B2C (личные, `app/`)

| URL | Папка | Кабинет |
|---|---|---|
| `/owner` | `app/owner/` | 👤 Кабинет владельца питомца |
| `/pethelper` | `app/pethelper/` | 🤝 Кабинет зоопомощника (волонтёра) |

## Кабинеты организаций B2B/B2C (`app/`)

| URL | Папка | Кабинет |
|---|---|---|
| `/org` | `app/org/` | 🏢 Кабинет организации (приют, клиника, магазин, заводчик...) |

Тип организации определяет набор активных модулей (permissions). Один кабинет — разные модули.

## Государственные кабинеты B2G (`app/`)

| URL | Папка | Кабинет |
|---|---|---|
| `/gov` | `app/gov/` | 🏛️ Кабинет государственного/контролирующего органа |

## Служебные кабинеты

| URL | Папка | Кабинет |
|---|---|---|
| `/admin` | `app/admin/` | ⚙️ Панель администратора платформы |

## Принцип разделения org vs orgs

- **`/orgs`** (plural) = публичный **список** всех организаций (REST-конвенция: plural для коллекции)
- **`/org`** (singular) = **рабочее пространство** конкретной организации (кабинет)

Этот принцип НЕЛЬЗЯ нарушать при добавлении новых страниц.


## Системный архитектурный принцип: базовые разделы + модули

Этот принцип применяется **ко всем кабинетам платформы** — организаций, владельцев, волонтёров, государственных органов, администраторов.

Каждый кабинет делится на:

- **Базовые разделы** — всегда присутствуют, определяют ядро кабинета
- **Подключаемые модули** — опциональные расширения, активируются пользователем

Модули хранятся в подпапке `modules/` внутри кабинета. Витрина модулей — `modules/page.tsx`. Каждый модуль — своя подпапка `modules/[id]/`.

### Пример: кабинет организации `/org/[orgId]`

**Базовые:** Главная, Организация, Сотрудники, Настройки, Модули (витрина)

**Модули:** Питомцы, Заявки, Аналитика, Документы...

### Порядок пунктов в сайдбаре (универсальный)

1. Базовые разделы (от общего к частному)
2. Активные модули (динамически)
3. Системные: Модули (витрина) → Настройки

### Как добавить новый модуль (в любом кабинете)

1. Создать `modules/[id]/page.tsx`
2. Добавить в `AVAILABLE_MODULES` в `modules/page.tsx`
3. Добавить таб условно в `layout.tsx` (`if (activeModules.includes('[id]'))`)
4. Добавить роут и URL-matcher в `layout.tsx`


## Типы кабинетов (Cabinet Types)

Кабинеты `org` и `gov` — это не монолиты, а **кабинеты с типом**. Тип определяет набор доступных модулей и их поведение.

### Типы org-кабинета

| Тип | Slug | Примеры |
|---|---|---|
| Приют | `org_shelter` | Муниципальный, частный, передержка |
| Ветеринарная клиника | `org_vet` | Клиника, кабинет, скорая |
| Зоомагазин | `org_shop` | Зоомагазин, корм, аксессуары |
| Заводчик | `org_breeder` | Питомник, razvedenie |
| Груминг | `org_grooming` | Груминг-салон |

### Типы gov-кабинета

| Тип | Slug | Примеры |
|---|---|---|
| Муниципалитет | `gov_municipality` | Администрация города/района |
| Управление ветеринарии | `gov_vetcontrol` | Госветнадзор |
| МЧС / Служба отлова | `gov_capture` | Городская служба отлова |

> Тип задаётся при регистрации и хранится в профиле организации. Этот slug используется в `module_cabinet_permissions` вместо просто `org` или `gov`.


## Платформенная система модулей (Module Marketplace)

Модули — переиспользуемые функциональные единицы (Питомцы, Заявки, Медкарты...). Их управление двухуровневое:

### Уровень 1 — Глобальный (`/admin`)

Администратор платформы определяет **совместимость** модуля с каждым **типом кабинета** (`org_shelter`, `org_vet`, `gov_vetcontrol` и др.). Если модуль не разрешён глобально — пользователь его вообще не видит.

### Уровень 2 — Локальный (Пользователь)

Если модуль разрешён глобально для данного типа кабинета — пользователь сам принимает решение его активировать или нет через витрину модулей `/modules`.

### Версионность модулей

Каждый модуль имеет версии (например, `1.0.0`, `2.1.0`). Пользователь может:
- Обновиться до актуальной версии
- Остаться на текущей (стабильной)
- Откатиться к предыдущей, если новая работает некорректно

Администратор помечает версии как `deprecated` с датой окончания поддержки.

### Схема таблиц БД

```sql
modules(id, slug, name, icon, current_version)
module_versions(id, module_id, version, changelog, is_stable, deprecated_at)
module_cabinet_permissions(module_id, cabinet_type, is_available)  -- cabinet_type = org_shelter, org_vet...
module_dependencies(module_id, depends_on_module_id)               -- зависимости
user_active_modules(id, user_id, module_id, pinned_version, activated_at)
```


### Зависимости между модулями

Модуль может **требовать** другой модуль как предварительное условие активации.

**Примеры зависимостей:**
- 🐾 `ОСВВ` (стерилизация/учёт бродячих) → требует `Питомцы`
- 📋 `Заявки на усыновление` → требует `Питомцы`
- 💉 `Вакцинация (партия)` → требует `Питомцы` + `Медкарты`
- 📊 `Аналитика поголовья` → требует `Питомцы`

**Правила:**
- При попытке активировать модуль с неудовлетворёнными зависимостями — показать предупреждение со списком недостающих модулей
- При деактивации модуля, от которого зависят другие активные модули — предупредить и показать список зависимых
- Зависимости хранятся в таблице `module_dependencies` и проверяются на бэкенде

### Варианты модуля (Module Variants)

Один модуль (`slug`) может иметь **разное поведение** в зависимости от типа кабинета. Это не отдельные модули, а *варианты* одного.

**Пример: модуль `pets`**
- `org_shelter` → полная таблица: `org_pet_number`, статусы, кураторы, хронология
- `owner` → упрощённый список «моих питомцев»
- `pethelper` → список подопечных с возможностью передачи

Реализация через `variant` prop в компоненте: `<PetsTable variant="shelter" />`. Конкретное решение принимается при имплементации каждого модуля.


## Общие UI-компоненты (`frontend/components/`)

UI-компоненты модулей хранятся в общей папке, а не дублируются по кабинетам:

```
frontend/components/
  pets/
    PetsTable.tsx       — таблица списка питомцев
    AddPetModal.tsx     — модалка добавления питомца
    PetCard.tsx         — карточка питомца
  requests/
  analytics/
```

Каждый `page.tsx` в кабинете только загружает данные из своего API и передаёт их в общий UI-компонент. Вся логика отображения — в компоненте.


## Принцип: один файл — один функционал

**Нельзя** записывать несколько несвязанных функциональных блоков в один файл. Каждый файл должен отвечать ровно за одну задачу.

- ✅ `AddPetModal.tsx` — только модалка добавления питомца
- ✅ `PetsTable.tsx` — только таблица списка питомцев
- ❌ `PetsPage.tsx` с модалкой, таблицей и фильтрами внутри одного файла на 500 строк

Если файл вырос больше ~200 строк и содержит несколько визуально/логически отдельных блоков — **разбить на несколько файлов**.
